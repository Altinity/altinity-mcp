Two questions decide everything: who is the OAuth Authorization Server to the client? (the broker axis) and how does MCP prove the user to ClickHouse? (the gating vs forwarding axis). They are independent.

Two independent axes — don't conflate them

① Front door — broker: true / false

Who acts as the OAuth Authorization Server to claude.ai / ChatGPT?

  • broker: true — MCP is the AS. It runs /authorize, /callback, /token and brokers the upstream IdP (CIMD inbound).
  • broker: false — MCP is a pure resource server. The external IdP runs the whole auth-code flow; MCP only validates the bearer.
② Back door — gating vs forwarding

How does MCP authenticate the user to ClickHouse? Auto-detected per endpoint, cached for the server's lifetime.

  • forwarding Authorization: Bearer <JWT> — CH validates the JWT natively (ClickHouse Antalya's token_processor).
  • gating Basic base64(email:JWT) — the ch-jwt-verify sidecar verifies per query (OSS CH).
Historically these were one knob (mode: gating|forward). Today they're orthogonal: the front door is config (broker); the back door is discovered at runtime (Bearer first, fall back to Basic on an auth error).

broker: true — MCP is the Authorization Server

Canonical mode. MCP fronts an IdP that clients can't talk to directly, using CIMD (no Dynamic Client Registration).

1
client → MCP
GET /oauth/authorize · client_id is an HTTPS CIMD URL · PKCE S256 · resource indicator
2
MCP
Resolves & caches the CIMD doc (redirect_uris, none / private_key_jwt). Mints its own upstream PKCE. Packs state into a JWE.
3
MCP → IdP
302 to Auth0 / Google /authorize with MCP's client_id + audience. User logs in & consents.
4
IdP → MCP
302 to /oauth/callback?code=…. MCP does not redeem yet — it wraps the upstream code into a 60s downstream JWE code.
5
MCP → client
302 to the client's redirect_uri with the downstream code.
6
client → MCP
POST /oauth/token · downstream code + PKCE verifier (+ private_key_jwt assertion for ChatGPT).
7
MCP → IdP
Now redeems the upstream code (client_secret + MCP's PKCE). Validates the returned id_token/access_token.
8
MCP → client
Returns the bearer access_token. No downstream refresh token — client re-authorizes on expiry.
Why defer redemption to step 7? The upstream IdP becomes the single cross-replica "used-codes" oracle — a replayed downstream code hits upstream invalid_grant. No shared state needed across MCP replicas.

broker: false — MCP is a pure resource server

Use when the upstream IdP natively supports CIMD. MCP exposes no /authorize·/callback·/token.

Only endpoint MCP serves

/.well-known/oauth-protected-resource (RFC 9728) — points authorization_servers at the upstream issuer. On 401 it returns WWW-Authenticate: Bearer … resource_metadata=….

Who does what

The IdP runs login, consent, code exchange & (if any) refresh. MCP never sees a client secret — it just verifies the presented JWT and forwards identity to CH.

CIMD — Client ID Metadata Documents

  • DCR is gone (#115). /oauth/register → 410 with an RFC 7591 error pointing at CIMD.
  • The client_id is itself an HTTPS URL to the client's published metadata.
  • MCP fetches & validates it each flow: HTTPS-only, no query/fragment, port 443, ≤5 KB, SSRF-guarded, LRU-cached.
  • token_endpoint_auth_method: none (PKCE-only — claude.ai) or private_key_jwt (signed assertion via jwks_uri — ChatGPT).
  • Metadata advertises client_id_metadata_document_supported: true.
Two independent PKCE pairs: client↔MCP and MCP↔upstream. State travels as a stateless JWE, so any replica can finish the flow.

JWT → ClickHouse: the wire format is auto-detected

On the first request to a CH endpoint, MCP probes — then caches the answer for the server's lifetime (per host:port).

MCP

Decode email from the unverified JWT payload (or namespaced */email).
4
probe
Try Basic base64(email:JWT) → cache Basic. (gating)
The email in the Basic username is just a hint so the sidecar knows who to check — the sidecar still verifies the JWT signature and rejects any mismatch between the signed email and the Basic user.

Bind the user in ClickHouse

CREATE USER "alice@example.com"
  IDENTIFIED WITH http
  SERVER 'ch_jwt_verify'
  SCHEME 'BASIC';

CH routes the Basic header to the ch_jwt_verify auth server; the name must match the user's email.

What the sidecar does, per query

  • Cryptographically verifies the JWT signature (JWKS).
  • Checks iss / aud / exp and the identity policy (e.g. require_email_verified, allowed domains).
  • Asserts the signed email == Basic username.
  • Source + helm chart live in altinity-oauth-helper (separate repo).
This replaced the old client-side cluster_secret approach — that config key is now removed, and the sidecar owns identity policy.

Forwarding mode — ClickHouse Antalya, native JWT

Antalya CH (token_processor since 25.8; latest 26.3) validates JWTs itself, so MCP just forwards the bearer — no sidecar, no Basic. Mainline CH lacks this, which is why forward mode targets Antalya / altinityantalya builds.

How MCP sends it

The bearer goes in HttpHeaders. Over HTTPS, MCP hands the token to clickhouse-go's GetJWT callback (so the driver can refresh it on reconnect). Auto-detect's first Bearer probe just succeeds.

What Antalya does

  • <token_processor> (e.g. google_oauth) verifies the signature against the IdP's JWKS.
  • The <token> user_directory mints an ephemeral CH user on the fly, with a preset role (e.g. oauth_demo_role).
  • No pre-created user, no email extraction, no Basic — identity comes straight from the verified JWT.

End-to-end — broker: true + gating (OSS)

The common production shape: MCP brokers the IdP, and CH delegates to the sidecar.

1
client ↔ MCP ↔ IdP
OAuth broker flow (slide 4). User signs in at Auth0 / Google; MCP returns a bearer JWT to the client.
2
client → MCP
Calls a tool (execute_query) with Authorization: Bearer <JWT>. MCP validates it.
3
MCP → CH
Auto-detect picks Basic → sends base64(email:JWT) to ClickHouse over HTTPS.
4
CH → sidecar
CH delegates to ch_jwt_verify; it verifies the signature & email, returns allow/deny.
5
CH → MCP → client
Query runs as alice@example.com with her grants; rows flow back to the client.
Same user identity end-to-end: the email claim signed by the IdP is the ClickHouse user — RBAC & row policies apply per person.
